一句話結論

rubydex 是「結構性查詢」的精準工具,但對 Rails 動態 / 文本場景幫不上,甚至會鎖死 Claude(case-1)。

以 recall-first 衡量,「不開 rubydex 的 Haiku」和「開 rubydex 的 GPT-5.5」並列最穩 —— 各只漏 1 個 hit。

而真正的「LLM 找不到」幾乎不存在 —— 錯誤集中在過報而非漏報。純 LLM 的程式碼搜尋能力被低估了。

01

怎麼測的

每個 case 對應一個具體可命名的 Ruby/Rails 撰寫手法(不是抽象的「找東西」)。每題跑兩輪,唯一差別是「能不能用 rubydex MCP」:

對照設計

with-rubydex:載入 rubydex MCP,限定只用語意工具(必要時可 Read)。
without-rubydex:換到獨立空目錄當 cwd,完全不載 MCP,只能 grep / Read。

兩版 prompt 只差第一行的工具指示,任務描述與回報格式完全相同 → 確保差異只來自「能不能用 rubydex」。每題結尾要求固定格式的 RESULT_SUMMARY 區塊,讓腳本自動抽數字比對。

判斷標準：Recall-first

搜尋任務裡,漏報(false negative)遠比過報嚴重 —— 漏一個 hit 等於 LLM 看不到該資訊、後續決策直接錯過(silent bug);過報只是增加 filter loading,不致命。

Ground Truth 是另外用 grep + 人工分類獨立精算的,不依賴受測跑次。

prompt 長相（示意，已匿名）

# with / without 只差這第①行：
請用 rubydex MCP 工具（不要用 grep，必要時可 Read）。   # with
請用 Grep / Read（不要呼叫任何 MCP 工具）。          # without

# ②任務（範例為虛構，非真實 case）：
列出所有真正呼叫到 Account::Profile#phone 的地方，分四類：
  A) 直接 profile.phone
  B) 透過 user.profile.phone（association traversal）
  C) 透過 presenter delegate（隱式呼叫）
  D) 透過 send(:phone) / public_send 動態呼叫
每點給 file:line + 屬於哪一類。

# ③回報格式（可被腳本抽取）：
RESULT_SUMMARY:
hits_total: <n>  hits_A: <n>  hits_B: <n>  hits_C: <n>  hits_D: <n>
END_SUMMARY

04

Recall-first 排名 — 漏越少越好

這是本報告最關鍵的圖。橫條 = 各條件在 14 題裡「不漏率」(hits ≥ GT 的題數);右側標注漏掉的 case 與總漏 hit 數。

最大反例：rubydex 工具陷阱（case-1）

has_one + delegate 鏈式是最戲劇的失敗:Claude 配 rubydex(Haiku-w、Opus-w)都拿 0 hits。Claude 看到 rubydex 可用就鎖死狂打 MCP,即使第一個 call 沒結果也不切回 grep。同題不開 rubydex 反而拿 25–26(接近正解 24)。rubydex 不只沒幫上忙,反而鎖死 LLM 不去用更有效的工具。

05

對稱誤差 & 成本效率

換成「對稱誤差」(過報漏報一視同仁)時,六條件「完全對」都落在 7–8/14,差不多 —— recall-first 才把差距拉開。token 為該條件 14 題的總 output token。

條件完成完全對diff ≤ ±10% 總 output token平均 / case

三個模型的角色

  • Opus 4.7 (low) — 最省 token(2.7k/case),但有「賭」屬性:case-8 用 44 token 命中,case-1 卻燒 2.1k 拿 0 分。
  • Haiku 4.5 — 最費 token(~6.4k/case),但 Haiku-grep recall 最穩(只漏 1)。
  • GPT-5.5 (codex) — token 居中(3.5–3.8k),最穩定無 outlier,但系統性過報。

Codex 與 Claude 的關鍵差異

  • 不踩鎖死陷阱 — case-1 配 rubydex 拿 26,它會 rubydex + grep 混用,不單押一邊。
  • 系統性過報 — 對 prompt 排除規則遵守度低(case-4/4A/7/10 普遍 +5~+27),但 recall 視角下「過報」是安全錯誤。
  • rubydex 對它影響小 — with/without 答案多半一致,更像「純加速器」而非「改變結果」。
06

結論：rubydex 何時該用

rubydex 的價值高度取決於「問題本質」。它是「結構性查詢」的好幫手、「文本式查詢」的負擔。

✓ 該用（結構性查詢）

  • constant / class / module 名引用 case-4 · 4A · 8
  • 多 namespace 同名 class 區分
  • 短形式 ↔ FQN 同 class 識別

✕ 不該用（文本 / 動態）

  • 字串 literal / symbol case-5 · 7 · 9 · 13
  • 動態組合 method 名 / class 名 case-2 · 12
  • 外部資源 yml / routes / locales case-5 · 10 · 11

對 Rails-heavy codebase(動態 DSL、symbol、yaml/routes),rubydex 幫不上的場景多於幫得上的。而扣掉「rubydex 鎖死」(case-1,工具問題)與「GT 口徑爭議」(case-13)兩個非能力因素後,沒有任何條件因為「找不到」而漏 —— LLM 的錯誤幾乎全是過報。純 LLM 的程式碼搜尋能力,沒有想像中弱。

關於匿名化 — 本報告所有 數字(hits / GT / token / cost / 排名)皆為真實測試結果。為保護受測來源,生產專案名一律以 App-A / B / C 表示,業務 class / 路徑已換成中性虛構名(保留結構難度、抽掉業務語意)。深入真實程式碼的逐題 ground-truth 推導不在公開範圍。
