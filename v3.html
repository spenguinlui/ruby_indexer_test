把 rubydex 當選擇器:
讓模型自己決定何時用它,有比較好嗎?

三代問法在變:V1 強迫模型用某個工具、V2 把 rubydex 丟進工具箱讓它自選、V3 = 一個會自己選工具的策略(先廣 grep,結構題才叫 rubydex 驗證,字串/動態盲區走 grep)。這次把 V3 selector 跑滿全 15 題 × 3 模型,跟 V2 的 grep-only 與 rubydex-available 放在同一把尺下重判(per-case 單一 judge)。問題很簡單:讓模型自己當選擇器,到底比「純 grep」或「光放工具箱」好不好?

一句話結論

給模型 rubydex —— 不管是「隨時可用」(V2)還是「叫它自己路由」(V3 selector)—— 在三個模型上都比純 grep 好;但「叫模型自己當選擇器」這條 prompt 捷徑只多加一點點,真正的槓桿在外層 orchestrator。

在同一把尺下重判全 15 題後,故事很乾淨:value 題品質三模型都 grep ≤ rubydex ≤ selector(Haiku 5.5→6.5→7.0、Opus 7.0→7.5→8.0、Codex 8.0→8.0→8.5 / 滿分 9)。Haiku/Opus 嚴格上升,Codex 強模型 grep 已接近天花板、rubydex 沒多加分但 selector 再頂高一格。rubydex 加分最集中在 namespace / constant 計數題(case-4、4A、8)—— 純 grep 會多算或漏短形式,rubydex 把數字釘準;這跟機器題的結構題發現完全一致。

但 selector 比「rubydex 隨時可用」只多贏一點點,而且 selector 實際呼叫 rubydex 又稀疏又沒對準題型:Haiku 在好幾個結構主場題(case-1/2)選了純 grep、rubydex 0 次,卻在字串盲區題叫了它。→ 「把 MCP 丟工具箱賭模型自己挑」不可靠,要靠外層 orchestrator 主動按題型路由(= V4)。

兩個 caveat:① case-7(polymorphic source_type silent bug)三個模型 9 格全摃龜 —— 都把 'income_expense' 當「對到真 model」,沒看穿 Rails 會 constantize 成不存在的頂層常數;這是推理盲點,rubydex 救不了。② 全程 n=1,小差異別當真。

03

rubydex 到底幫多少?越弱的模型幫越多

9 個 value 題的判讀品質分數(ok=1 / part=.5 / bad=0,滿分 9),三條件並排。三模型都 grep ≤ rubydex ≤ selector,增幅隨模型變強縮小到 Codex 上 rubydex 跟 grep 平手(grep 已 8/9 接近天花板)、但 selector 仍再頂高一格。

6 機器題的平均 file-recall 也同向但更接近天花板:Haiku .82→.93→.94、Opus .91→.93→.89、Codex .94→.97→.93。注意:機器題上「rubydex 隨時可用」(中欄)其實常是最高的 —— selector 的路由在強模型上偶爾反而少叫 rubydex、小輸一截(Opus case-13、Codex case-1)。

✓ rubydex 一致加分的題型

  • namespace / constant 計數(case-4、4A、8):純 grep 會漏短形式或多算,find_constant_references 把數字釘準 —— 三模型皆然。
  • rename / 關聯影響面(機器 case-14、2):rubydex 直接給引用集,recall 不掉還少讀檔。

✗ 誰都救不了的題型

  • 字串 / 動態推理(case-7、10、3):答案不在靜態結構,rubydex 純 overhead。
  • case-7 全摃龜:9 格都漏 income_expense silent bug。Opus 最慘(三條件全 bad)—— 自信地把 4 個 bug 全「解釋成正常」。
04

關鍵問題:selector 真的「會選」嗎?

selector 的賣點是「按題型自己路由 rubydex」。實際看它每題叫了幾次 rubydex(V3-selector arm)—— 如果路由有效,應該結構題多叫、盲區題不叫。結果不是。

case型Haiku rdxOpus rdxCodex rdx
路由沒對準

看 結構 列本該多叫 rubydex,但 Haiku 在 case-1/2/14 多次 0 次(挑了純 grep);反而 盲區 的 case-7 字串題 Opus 叫了 14 次、Codex 18 次(rubydex 對字串根本沒用)。Haiku 全 15 題只 4 題真的用 rubydex、Opus 7 題且多是 1 次。→ prompt 層的「選擇器」是模型隨興挑,不是按題型路由。這正是為什麼 selector 只比「rubydex 隨時可用」多贏一點點。

真正能兌現 rubydex 價值的,是把路由從「拜託模型自己挑」搬到 外層 orchestrator:先判 intent → 結構題才召喚 rubydex → 多路召回打分 → 只把 top-k 餵進 LLM。本輪用 prompt 近似驗證了「rubydex 對結構題有料」,但也證明 prompt-selector 這條捷徑不可靠 —— 那是 V4(orchestrator + ruby-lsp-rails / rails runtime)的事。

05

讀檔 / token:結構題上 rubydex 能大砍 context

6 機器題的 context token(in+cache+out,K)。rubydex 在結構題讓模型免翻檔 → context 大降;盲區題則是白付 MCP 載入稅。旗艦是 case-14 rename 影響面。

case-14(Haiku):grep 讀 17 檔 / 659K → selector 讀 2 檔 / 223K(−66%),recall 都 1.0。find_constant_references 一次給影響集、不用翻檔。但盲區題相反:case-13(Haiku)grep 428K → selector 1231K(載 MCP + 多讀);case-6 grep 96K → selector 340K。→ 又是 intent 路由:只結構題開 rubydex 才划算。

06

勘誤與方法

這輪查證時抓到的問題,寫出來才誠實。

① answer key 修正:case-12 ledger 型別 1 → 4

舊 key 寫 ledger_inferred_types=1。但讀 ledger_factory.rb 的 live code:LedgerFactory#steps 回的是 Success(user_ledger) = build_ledger_by_user(...) 的結果,依「是否換約 × 是否含稅」回 4 種型別(2 UserLedger + 2 Decorator);key 寫的那個 Ledger 只是傳進去的內層參數,從不直接回傳。→ 正解是 4,答 4 的模型是對的;key 已修。

② case-7 確認 key 沒錯:income_expense 真的是 silent bug

有人(模型)說 income_expense 對到 …FixedCharge::Models::IncomeExpense 所以不是 bug。但它存進 Rails polymorphic 關聯,讀 .source 時 Rails 會 constantize 頂層 IncomeExpense(不存在)→ 仍是 silent bug。namespaced 那個 model 構不到。→ key 正確,是模型判錯。

③ answer key 修正:case-4 / case-4A 「扣 class def」是 bug(2026-05-26 全 key 重驗)

為了把整批 key 釘乾淨,把剩下 7 個沒驗過的 key 對 live code 重驗了一輪。發現 case-4 / case-4A 的計算公式裡有個共同 bug:扣掉了「class def 自身」那一行,但實際上 class Invoice < ApplicationRecord 那行不含 FQN 字面、不在 raw count 內,本來就不該扣。修正後:

  • case-4:total 101 → 102(+1)。原本答 102/106 的 LLM 其實更接近正解;判讀無變化。
  • case-4A:per-class +1,total 141 → 145(+4)。Opus/Codex 的純 grep 結果(150/151)從「過報 ~9」變「過報 ~5-6」→ recall-first 下從 part 升到 ok。**這讓 Codex 的 grep≤rubydex 變成平手(8.0=8.0),不是嚴格 < —— 不過 selector 仍勝**。
  • 其他 5 個 key(case-3/8/9/10/11)逐項對 live code,全部 ✅ matches,沒改。
方法:為什麼要「per-case 重判」

第一輪我為了平行,每個模型派一個 judge,結果不同 judge 對同一份 key 套出不同嚴格度(case-10:某 judge 全 bad、另一個全 ok)→ 跨模型分數被污染。改成每題一個 judge 判該題全 9 格後,同題內跨模型/跨條件才真的可比。本報告主表用的是重判後的乾淨資料。

07

結論

① 給模型 rubydex 是淨正面:value 題品質三模型全部 grep < rubydex < selector,從沒讓誰變差;加分集中在 namespace/constant/rename 這類結構題,跟機器題發現一致。

② 越弱的模型幫越多(Haiku +1.5、Opus +1.0、Codex +0.5 / 滿分 9):強模型純 grep 已接近天花板,rubydex 的邊際價值遞減。

③ 「selector」這條 prompt 捷徑只多贏一點點,而且路由不可靠:模型實際叫 rubydex 又稀疏又沒對準題型(結構題不叫、盲區題猛叫)。selector ≈ rubydex-available。

④ 有些題誰都救不了:case-7 的 polymorphic silent bug 是推理盲點,任何工具都沒幫上;Opus 還因為太會「講道理」而最慘。

⑤ 所以 V4 方向明確:價值在 rubydex 對結構題的料,但要靠外層 orchestrator 主動按 intent 路由來兌現,不是把 MCP 丟工具箱賭模型自己挑。下一步:接 ruby-lsp-rails / rails runtime + 多路召回打分。

如何看這些數字

每格只跑一次(n=1),單次變異很大。所有結論都是方向性訊號;單題的小差異(尤其 ±1 個 part/ok)別當定論。要可發布的強主張需每格 N 次取中位。
