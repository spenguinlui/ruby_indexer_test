⚠

勘誤:原 V2 的「不載 MCP」其實偷載了 MCP

先把這件事講清楚 —— 這也是這份報告結論方向改變的原因。我們把它寫出來,是希望別人少踩同一個坑。

問題

原 V2 的 with/without 設計是「cwd 不同、不另外載 MCP」:without 變體把工作目錄切到 ~/ruby_indexer/control_room(一個空目錄),以為這樣就不會載入 rubydex。但兩個 CLI 都會自動往上找設定檔 —— Claude Code 從子目錄 auto-discover 上層的 .mcp.json、codex 則載入全域註冊的 ~/.codex/config.toml。結果 without 變體照樣把 rubydex 載進去、也真的呼叫了它。

翻 transcript 才發現:三個模型的 without 輪都實際用到了 rubydex。例如某題的 without 輪呼叫了 rubydex 達 20 次。所以原報告寫的「without = 無 MCP / grep-only」「with/without 只差載不載 MCP」並不成立 —— 它量到的其實是「MCP + cwd-A」對「MCP + cwd-B」,從來沒有乾淨的「有 MCP vs 無 MCP」。

為什麼這會讓結論失真

被污染的是對照組(without)。without 偷用了 rubydex,等於把「純 grep」的基線墊高了;於是 with(本來就有 rubydex)很難贏過它 —— rubydex 看起來「沒幫助、甚至偏負」,其實是在跟「另一個也有 rubydex 的自己」比。把 without 修乾淨、基線掉回真實高度後,with 在好幾題就明顯贏回來了。

修法

  • Claude:加 --strict-mcp-config —— 只用 --mcp-config 明確指定的 server,忽略 auto-discovery 與全域設定。with = strict + 載 4 個 rubydex;without = strict + 不給任何 config(真零 MCP)。
  • codex:without 改用 --ignore-user-config,不載 ~/.codex/config.toml(舊版用 -c 'mcp_servers={}' 無效,全域 server 照載)。

已驗證乾淨

  • 修後重跑 codex without 輪:15 題全部 0 次 rubydex 呼叫(逐題核對 jsonl)。
  • Claude without 輪:strict 模式下連 rubydex 的 tool schema 都不進 context。
  • 三個模型(Haiku / Opus-low / GPT-5.5)全部用修好的 runner 重跑,以下圖表全是乾淨數據。

這是這份修訂版唯一動到的東西:對照組的環境隔離。prompt、題庫、計分方式都沒變。原始的污染數據我們留著當「control_room 會自動載 MCP」的證據,沒有丟。

01

V2 跟 V1 差在哪

三項刻意改動,目的是把測試從「人工強迫」推向「真實使用」,以提高測試準度。

① prompt 最大模糊化

V1 會把答案形狀餵給模型(給 FQN、def 位置、引用形式、排除規則、固定 RESULT_SUMMARY)。V2 只給最小語意錨點(短名),不給路徑、不給形式、不給排除 —— 逼模型自己探索。
例外:case-4/4A/8 是刻意設計的「prompt 明確度」對照三胞胎,保留 V1 明確度,否則三題塌成一題。

② 開放全部工具(含 Glob)

V1 用文字「請用 Grep / Read」把 Glob 排除掉了。V2 不寫任何工具限制 —— Grep / Glob / Read / Bash 全開,模型自選。

③ with/without 只差「載不載 MCP」

兩版 prompt 文字完全相同。差別純在環境:with 載入 rubydex MCP、without 不載。量到的是最有決策價值的問題 ——「把 rubydex 放工具箱、不強迫用,到底有沒有讓結果變好」。
(修訂)原本靠「切 cwd 到空目錄」來不載 MCP,但 CLI 會自動往上找設定檔導致 without 偷載 rubydex;已改用 --strict-mcp-config / --ignore-user-config 做真正的隔離,見勘誤。

④ 新增 refactor 影響面 case

對標官方唯一沒涵蓋的「真的改 code」維度。改問「rename 一個 class 的完整影響清單」,對 ground-truth 影響集算 recall(不跑測試、不依賴 DB)。

prompt 長相對比（示意，已匿名）

同一題(找全會真正呼叫到某個「被委派出去的屬性」的地方),V1 把答案形狀都講白了;V2 只留一句自然語句。

# ── V1（詳細）：給 FQN + def 位置 + 4 種分類 + 固定回報格式 ──
請用 rubydex MCP（不要用 grep）。   # with     請用 Grep / Read（不要用 MCP）。   # without
目標 method：Account::Profile#phone（定義在 .../profile.rb:NN）
列出所有真正呼叫到它的點，分四類：
  A) 直接 profile.phone   B) user.profile.phone（關聯）
  C) presenter delegate（隱式）   D) send(:phone) 動態
RESULT_SUMMARY:
hits_total: <n>  hits_A: <n>  hits_B: <n>  hits_C: <n>  hits_D: <n>

# ── V2（模糊）：一句話。不給 FQN / 路徑 / 形式 / 排除 / 回報格式 ──
# with 與 without 的 prompt 文字「完全相同」，只差執行時載不載 MCP
在 App-A 這個專案裡，Account 的 profile 上有個 phone 欄位。
我想知道整個專案裡，實際上有哪些地方會真正呼叫到這個 phone？
幫我找全，每個給 file:line。

refactor case 的 V2 prompt（同樣只給一句,連改名對象都只點到 class、不給檔案):

在 App-A 裡，我打算把某 domain 下的一個 repository class 改名。
動手前我想先拿到完整的「改名影響清單」：這次改名總共要動到哪些檔案、哪些行？
每個給 file:line。漏一個都可能讓改名後在執行期壞掉，所以請盡量找全。

真實 prompt 用各專案的中文業務語句與絕對路徑;此處比照 V1 換成中性虛構名(Account::Profile#phone 等),保留「給多少資訊」的結構,抽掉業務語意。

為什麼想換成自然選用

「強迫用 rubydex」適合測「它能做到什麼」,但真實開發者不會被強迫用某個工具。V2 的自然選用,想看的是另一個問題:「裝了 rubydex 之後,日常使用實際會發生什麼」。兩種設定互補,不互相取代。

「+rubydex」= MCP 載入但模型自己決定用不用;「grep」= 真正不載 MCP(已修正污染)。每格底色深淺=recall 高低(越紅漏越多);「+rubydex」欄加綠框 = 該模型 with > without(rubydex 幫到 recall)、加紅框 = with < without(反而漏更多)。

觀察(修正污染後):rubydex 提升 recall 的次數比原測明顯,但仍混合

18 組 with/without 對照中,rubydex 提升 recall 的有 5 組(原污染版只看到 1):case-1 的 Haiku(0.57→0.79)與 Codex(0.64→1.00)、case-2 的 Haiku(0.83→0.92)與 Opus(0.83→0.92)、case-13 的 Haiku(0.50→1.00)。拉低的有 2 組:case-2 Codex(1.00→0.83)、case-14 Haiku(1.00→0.88)。其餘 11 組打平(多在強模型端 + case-5/6 兩邊都滿分)。

為什麼跟原測差這麼多?因為被墊高的對照組被拉回來了。最戲劇化的是 case-1:原污染版顯示「Opus 1.00→0.64,rubydex 害的」,乾淨重跑後 Opus 兩邊都是 0.64(打平),而真正的故事是 純 grep 會漏掉 spec 檔,rubydex 反而幫 Haiku、Codex 補回去。見深潛。

一個正面發現:模型會自己判斷要不要用 rubydex

放開限制後,在 rubydex 看不到的題(source_type 字串、callback symbol、i18n yaml),模型的「+rubydex」那輪多半沒去呼叫 rubydex —— 自己改走 grep / bash。這也補充了 V1 的觀察:V1 看到的「LLM 傾向過度呼叫 rubydex」,比較像是「強迫用」設定下的產物;自然選用時,模型多半只在 rubydex 的強項題(多 namespace、常數解析)才會用它。

04

case-1 深潛:污染如何把結論整個翻過去

has_one + delegate 鏈式(找全會碰到某個被委派屬性的點)—— V1 最受關注的一題,也是這次污染修正後翻最大的一題。這題很適合用來看「對照組被污染」會怎麼誤導結論。

污染版(原 published):看起來 rubydex 害了 Opus

條件(污染版)recall當時的解讀
Opus · grep(以為不載 MCP)1.00「純 grep 找全」—— 但其實這輪偷用了 rubydex + grep
Opus · +rubydex0.64「載了 rubydex 反而漏 5 個 spec」→ 誤判成 rubydex 有害

當時把它寫成「rubydex 害 Opus 從 1.00 掉到 0.64」。但那個 1.00 的基線是偷用 rubydex 撐出來的 —— 對照組根本不乾淨。

乾淨重跑:真正的 grep-only 會漏 spec,rubydex 反而幫忙補回

Haiku0.790.57純 grep 漏掉 5 個 spec + client.rb;rubydex 幫補回

怎麼解讀

把對照組修乾淨後,case-1 的真相跟原報告相反:純 grep(真正不載 MCP)會漏掉一批 spec 檔(它們透過關聯間接碰到目標屬性,字面比對掃不到),rubydex 反而幫 Haiku、Codex 把這些補回來(Codex 0.64→1.00)。Opus 則兩邊都 0.64,打平。原報告那句「rubydex 害 Opus 從 1.00 掉到 0.64」完全是污染造成的假象 —— 那個 1.00 是 without 偷用 rubydex 撐出來的。這也是為什麼整份報告的結論方向要改。

提醒:這是 n=1。case-1 三個模型的方向一致(grep 漏 spec、rubydex 幫補),算是比較穩的訊號;但單格的小數點別當精確值看。

新增:refactor 影響面 case

官方 benchmark 唯一我們 V1 沒涵蓋的維度 —— 真的去改 code 的 refactor。官方用「改完測試通過」當品質訊號;我們改問「rename 一個 class 需要動到的完整影響清單」,對 ground-truth 影響集算 recall(不跑測試、不依賴 DB,更直接量到爬 code 完不完整)。

模型(乾淨)+rubydexgrep(真無MCP)漏了什麼

強模型滿分、有無 rubydex 都一樣 → 這題 rubydex 無加值。Haiku 是這 6 題裡 rubydex 唯二拖累的格之一(1.00→0.88):載了 rubydex 後它偏向走語意查詢、少做 grep 廣掃,於是漏掉兩個 spec 裡的 字串引用(instance_double('...') 那種字串非常數,語意 indexer 本來就不認,得靠 grep 才掃得到)。純 grep 反而把它們都掃到了。這跟 case-1 是同一個機制的兩面:rubydex 在場時 Haiku 容易少 grep —— 在 case-1 那題剛好幫上忙、在這題剛好幫倒忙。

對照官方

官方「測試通過」訊號無法分辨「漏掉的引用剛好沒被測試覆蓋」;本 case 的 file-level recall 直接抓出這種漏 —— 這是把官方那一格納進我們矩陣後,還能補上的盲點。

07

結論:自然選用下,該不該裝 rubydex

✓ 看到 recall 加分的場合

  • 純 grep 會漏的間接 / 關聯引用(spec 透過關聯碰到目標)case-1
  • 裸短形式 ↔ FQN 解析、常數真偽判定 case-4 · 7(Opus)
  • 多 namespace 同名 class 區分 case-4A
  • 弱模型(Haiku)在 hash dispatch 類題,rubydex 把漏的補回 case-13 0.50→1.00

△ 仍需留意 / 偶爾倒扣

  • 還是混合結果:18 組裡 5 幫、2 傷、11 平,不是穩定全贏
  • 較小的模型(Haiku):rubydex 在場時有時少做 grep、漏掉 spec 字串 case-14 · 2(Codex)
  • Opus 端成本仍 +48%;每格 n=1,變異大,別過度解讀單一數字

先講最重要的:原 V2 報告的核心結論(「rubydex 中性偏負、純 LLM 被低估」)是建立在一個被污染的對照組上,已作廢。修掉污染、把「真正不載 MCP」的基線拉回真實高度後,rubydex 對 recall 的幫助比原測明顯不少 —— 18 組對照裡提升的從 1 組變成 5 組,最清楚的是 case-1(純 grep 漏 spec、rubydex 補回)。

但也別過度修正成另一個極端:這仍是混合結果、變異很大(每格只跑一次)。rubydex 並非穩定全贏 —— 還是有 2 組被它拖低(多半是弱模型載了 rubydex 後少做 grep 廣掃、漏掉 spec 字串)。官方公布的 30–47% 提升是在它最擅長的任務(IRB 常數 rename)上量到的;本實驗補充的是「放寬到 15 種真實 Rails 手法 + 自然選用」這個角度,結論是:rubydex 對 recall 確實有正貢獻,集中在 grep 的結構性盲區(間接引用、短形式 / 常數解析),但不是無腦全掛就穩賺。

小提醒

這些結論只在「本題庫 + 自然選用 + 這三個模型 + n=1」的範圍內成立,屬於 V1 的延伸觀察,不是對 rubydex 或官方 benchmark 的否定。我們也很感謝這次的污染 bug 逼我們把對照組做乾淨 —— 想提醒同樣在做 MCP benchmark 的人:「不載 MCP」的對照組要確認 CLI 沒從上層目錄或全域設定偷載進來。不同題型、不同用法很可能得到不同結果,歡迎在自己的 codebase 上試。
