rubydex 帶來多少差別?

V1 是「強迫用 rubydex vs 強迫用 grep」的對照。V2 換成更貼近日常開發的設定:prompt 寫得很模糊、開放全部工具(含 Glob)、with/without 只差載不載入 MCP —— 由模型自己決定要不要呼叫 rubydex。另外補一個對標官方 benchmark 的 refactor 影響面 case。這是 V1 的延伸觀察,不是要否定誰。

一句話結論

在自然選用下,把 rubydex 加進工具箱,對 recall 多半是中性、偶爾偏負;成本則有增有減,影響在較小的模型上比較明顯。

01

V2 跟 V1 差在哪

三項刻意改動,目的是把測試從「人工強迫」推向「真實使用」,以提高測試準度。

① prompt 最大模糊化

V1 會把答案形狀餵給模型(給 FQN、def 位置、引用形式、排除規則、固定 RESULT_SUMMARY)。V2 只給最小語意錨點(短名),不給路徑、不給形式、不給排除 —— 逼模型自己探索。
例外:case-4/4A/8 是刻意設計的「prompt 明確度」對照三胞胎,保留 V1 明確度,否則三題塌成一題。

② 開放全部工具(含 Glob)

V1 用文字「請用 Grep / Read」把 Glob 排除掉了。V2 不寫任何工具限制 —— Grep / Glob / Read / Bash 全開,模型自選。

③ with/without 只差「載不載 MCP」

兩版 prompt 文字完全相同。差別純在環境:with 載入 rubydex MCP、without 不載。量到的是最有決策價值的問題 ——「把 rubydex 放工具箱、不強迫用,到底有沒有讓結果變好」。

④ 新增 refactor 影響面 case

對標官方唯一沒涵蓋的「真的改 code」維度。改問「rename 一個 class 的完整影響清單」,對 ground-truth 影響集算 recall(不跑測試、不依賴 DB)。

prompt 長相對比（示意，已匿名）

同一題(找全會真正呼叫到某個「被委派出去的屬性」的地方),V1 把答案形狀都講白了;V2 只留一句自然語句。

# ── V1（詳細）：給 FQN + def 位置 + 4 種分類 + 固定回報格式 ──
請用 rubydex MCP（不要用 grep）。   # with     請用 Grep / Read（不要用 MCP）。   # without
目標 method：Account::Profile#phone（定義在 .../profile.rb:NN）
列出所有真正呼叫到它的點，分四類：
  A) 直接 profile.phone   B) user.profile.phone（關聯）
  C) presenter delegate（隱式）   D) send(:phone) 動態
RESULT_SUMMARY:
hits_total: <n>  hits_A: <n>  hits_B: <n>  hits_C: <n>  hits_D: <n>

# ── V2（模糊）：一句話。不給 FQN / 路徑 / 形式 / 排除 / 回報格式 ──
# with 與 without 的 prompt 文字「完全相同」，只差執行時載不載 MCP
在 App-A 這個專案裡，Account 的 profile 上有個 phone 欄位。
我想知道整個專案裡，實際上有哪些地方會真正呼叫到這個 phone？
幫我找全，每個給 file:line。

refactor case 的 V2 prompt（同樣只給一句,連改名對象都只點到 class、不給檔案):

在 App-A 裡，我打算把某 domain 下的一個 repository class 改名。
動手前我想先拿到完整的「改名影響清單」：這次改名總共要動到哪些檔案、哪些行？
每個給 file:line。漏一個都可能讓改名後在執行期壞掉，所以請盡量找全。

真實 prompt 用各專案的中文業務語句與絕對路徑;此處比照 V1 換成中性虛構名(Account::Profile#phone 等),保留「給多少資訊」的結構,抽掉業務語意。

為什麼想換成自然選用

「強迫用 rubydex」適合測「它能做到什麼」,但真實開發者不會被強迫用某個工具。V2 的自然選用,想看的是另一個問題:「裝了 rubydex 之後,日常使用實際會發生什麼」。兩種設定互補,不互相取代。

02

Recall 主表 — 6 個可機器計分的題

這 6 題答案是「一組檔案 / 位置」,可自動算 recall(= 找到的 ground-truth 檔 / 應找到)。每格是該條件的 recall;綠=滿分,越紅漏越多。粉紅底=「有 rubydex 反而比沒有低」(recall-first 下值得留意的格)。

case Haiku
+rubydex		Haiku
grep		 Opus
+rubydex		Opus
grep		 Codex
+rubydex		Codex
grep

「+rubydex」= MCP 載入但模型自己決定用不用。粉紅底格 = with < without(rubydex 在場反而漏更多)。

觀察:這個題庫裡,rubydex 對 recall 多半中性、偶爾偏負

18 組 with/without 對照中,rubydex 幾乎沒有提升 recall(只有 1 個例外:codex case-1,0.57→0.64)。少數情況還會略低:Haiku case-1/2、Opus case-1;其餘多半打平(case-5/6/14 在強模型端)。也就是說,在這組「找全」任務上,把 rubydex 放進工具箱沒看到 recall 上的正貢獻 —— 這跟它的設計強項(結構性常數查詢)不在同一類題有關。

Opus 端整體貴約 53%,在這些題沒看到對應的 recall 提升;比較明顯的單題是 Opus case-1(成本近一倍、recall 從 1.00 到 0.64)。不過成本是有增有減的 —— 有些題 with 反而便宜(rubydex 直接給答案、省掉 grep 探索)。整體看,recall 端比較一致地中性偏負,成本則因題而異。

一個正面發現:模型會自己判斷要不要用 rubydex

放開限制後,在 rubydex 看不到的題(source_type 字串、callback symbol、i18n yaml),Haiku 與 Opus 的「+rubydex」那輪呼叫 rubydex 的次數是 0 —— 它們自己改走 grep / bash。這也補充了 V1 的觀察:V1 看到的「LLM 傾向過度呼叫 rubydex」,比較像是「強迫用」設定下的產物;自然選用時,模型多半只在 rubydex 的強項題(多 namespace)才會用它。

04

case-1 深潛:從「卡住」到「偏食」

has_one + delegate 鏈式(找全會碰到某個被委派屬性的點)—— V1 最受關注的一題,V2 的表現更微妙、也更貼近真實使用。

條件recall行為
V1 · 強迫 rubydex(Haiku/Opus)0 / 24被要求只用 MCP、沒有 fallback grep → 卡在 rubydex 上、幾乎全漏
V2 · Opus · grep(不載 MCP)1.00純 grep,找全(含 spec 檔)
V2 · Opus · +rubydex0.64叫了 rubydex 2 次 + grep 3 次,漏掉 5 個 spec 檔
怎麼解讀

自然選用化解了 V1 那種「卡住全漏」(0 → 0.64,模型會混用 grep 了);不過 rubydex 在場時,recall 還是略低於純 grep 的基線 —— 它讓模型偏向語意結果、少做一點 grep 廣掃,於是漏掉 spec 那種要靠 grep 才掃得到的角落。對 recall-first(把漏報看得比較重)來說,這會略為扣分。

05

新增:refactor 影響面 case

官方 benchmark 唯一我們 V1 沒涵蓋的維度 —— 真的去改 code 的 refactor。官方用「改完測試通過」當品質訊號;我們改問「rename 一個 class 需要動到的完整影響清單」,對 ground-truth 影響集算 recall(不跑測試、不依賴 DB,更直接量到爬 code 完不完整)。

模型+rubydexgrep漏了什麼
Haiku0.820.82定義檔本身(連帶自動載入的檔名連動)+ 2 個測試檔裡的字串引用
Opus-low1.001.00—
Codex1.001.00—

強模型滿分、有無 rubydex 都一樣 → 這題 rubydex 無加值。Haiku 兩邊都 0.82,漏的正是兩個鑑別點:autoload 檔名連動(改名要連檔名一起改,否則載入炸)與 測試檔字串引用(字串非常數,語意 indexer 也不認)。

對照官方

官方「測試通過」訊號無法分辨「漏掉的引用剛好沒被測試覆蓋」;本 case 的 file-level recall 直接抓出這種漏 —— 這是把官方那一格納進我們矩陣後,還能補上的盲點。

這 9 題答案不是「檔案清單」而是「值 / 數量 / 推論」,無法機器算 recall,改逐筆讀輸出對正解判:✓ 命中關鍵值 / ~ 部分 / ✗ 漏或錯。

value 題的三個結論

1 · rubydex 幾乎不改變正確性

9 題裡 8 題 with≈without。唯一明確加分是 case-4 的 Haiku:寬鬆 prompt 下純 grep 把短形式漏光(0),rubydex 幫它抓回全部 —— 對的是「弱模型 + namespace 短形式 + 該用 rubydex」這精確組合。

2 · 正確性主要看模型強弱

Opus / Codex 在難題(source_type boss、字串引用、反推 controller#action)穩;Haiku 偏弱 —— 但弱在模型本身,不是 rubydex(這些題 Haiku 多半根本沒叫 rubydex)。

3 · V1 的幾個狀況沒再出現

class_eval 的「太信嚴格 AST 而答 def 不存在」、case-1 的「卡住」—— 在自然選用 + 模型會真的去讀原始碼之後,V2 裡都沒再看到。

07

結論:自然選用下,該不該裝 rubydex

✓ 值得用的場合

  • 多 namespace 同名 class 區分 case-4 · 4A
  • 短形式 ↔ FQN 識別,且配弱模型 + 明確指示
  • 把它當「加速器」而非「改變結果」(強模型端)

△ 幫助有限 / 需留意

  • 純丟著讓模型自選 —— recall 多半中性、偶爾偏負 case-1 · 2 · 13
  • 較小的模型(Haiku):rubydex 在場時容易少做 grep、漏掉 spec
  • 文本 / 動態 / 外部資源 —— 模型通常會自己跳過 rubydex

官方公布的 30–47% 提升,是在 rubydex 最擅長的任務(IRB 常數 rename)上量到的,很有參考價值。本實驗想補充的是另一個角度:把題庫放寬到 15 種真實 Rails 手法、再讓模型自然選用後,這樣的優勢在我們的題庫裡就沒那麼普遍。rubydex 的價值比較集中在「對的題(多 namespace 常數)+ 對的模型 + 明確指示」;若只是默認全掛、丟著讓它自選,在這個題庫沒看到 recall 上的好處,Opus 端成本還高一些。反過來說,放開 grep / Glob / bash 之後,純 LLM 的程式碼搜尋能力,其實比想像中更穩。

小提醒

這些結論只在「本題庫 + 自然選用 + 這三個模型」的範圍內成立,屬於 V1 的延伸觀察,不是對 rubydex 或官方 benchmark 的否定。不同題型、不同用法很可能得到不同結果 —— 歡迎在自己的 codebase 上試。

關於匿名化 — 所有 數字(recall / 成本 / 工具呼叫)皆為真實測試結果。為保護受測來源,生產專案以 App-A / B / C 表示,業務 class / 路徑換成中性虛構名(保留結構難度、抽掉業務語意)。V2 與 V1 報告 用同一套匿名規則。
