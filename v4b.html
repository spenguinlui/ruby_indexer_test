一句話結論

V4-β.2(orchestrator + rails-dsl)不是「下一代 V4」,是有強烈題型 × 模型相依的 selective add-on。對 Codex 強模型而言:rails-dsl 把它變成 machine-recall 完美機(0.857→1.000、ctx −30%),代價是 value 大降(8.5→6.0,計數與 silent bug 推理題崩);對 Opus 個案性勝(case-7 silent bug bad→ok);對 Haiku 雙降,連最該贏的主場 case-11 都因 1-shot 答案後撞 cwd 衝突直接拒絕作答。

最戲劇的反諷是 Haiku case-11 routes-helper(rails-dsl 的指定主場題):context 從 1209K 砍到 80K、rdx 呼叫從 19 變 0,理論上是 β.2 該大勝的場景。但 Haiku 拿到 rails-dsl 一次給回的完整答案後,撞到「cwd（工作目錄）在 /Users/liu/ruby_indexer、但 project_root 在 /Users/liu/sunnyfounder/e_trading」的工具路徑衝突,直接拒絕作答 → bad。給太多 high-confidence 上下文,弱模型反而被看似 instruction 矛盾的訊號搞亂。

強模型的二次 trade-off:Codex machine 全綠(case-1 delegate 鏈 recall 0.64→1.00 +36pp、ctx −45%),但同樣的「rails-dsl 給高 confidence Rails 知識」上下文,在 case-7 polymorphic silent bug 題上讓 Codex 把 'income_expense' 解讀成 ElectricityAccounting::ChargesCalculation::FixedCharge::Models::IncomeExpense(named with high confidence)→ explain-away（用旁證合理化掉真正的 bug）silent bug → part → bad。在 case-4/4A 計數題上 total 從 ~100/145 暴跌到 41/58(−60%),疑似 top_k truncation（候選列表被截斷）或 false-confidence「我都看完了」early termination。

三個 caveat:① 全程 n=1,單次變異大,所有結論都是方向訊號。② β.2 的 rails-dsl 對「Rails-aware 但非 Rails-knowledge 黑洞」題型(case-8/10 Haiku 略升)有閃光,但 main path(case-1/13)反而受傷。③ Opus 的 case-7 win 是真的、但同模型 case-10 也從 ok 退到 part —— 個案性 win 不能歸納成「Opus 適合 β.2」,需要更多 N。

01

從 V4-α 到 V4-β.2:為什麼加 rails-dsl backend

V4-α 把 routing 拉到外層 MCP 是對的方向,但只有 rubydex(結構 AST)+ ripgrep(字串)兩個 backend,沒兌現「多工具協同 = 各擅其場」的承諾 —— 像把人請來開店、只擺了兩個架。β.2 補的第三個 backend 是 rails-dsl。

rails-dsl backend 是什麼

靜態 prism AST 掃 Rails 慣用 DSL,索引兩類資訊:

  • Routes:resources / get / post / namespace 等宣告,索引「URL helper ↔ controller#action ↔ http verb + path」三向映射。
  • ActiveRecord 關聯:has_many / belongs_to / has_one,特別標出 polymorphic: true 跟 class_name: '...' 顯式對應。

另外搭配兩個輔助升級:tokenizing query scorer(對 query 分詞、跟 backend 候選作 token-level 相關性排序,而不是純名稱字面)、hint vocabulary(把 route / association / polymorphic / dsl 等 hint 接上 backend dispatch)。

為什麼選 rails-dsl 而不是其他

backend 候選能補的盲區實作難度選了沒
rails-dsl(靜態)routes 反查、polymorphic 關聯、class_name string中(prism AST + 慣用 DSL 樣板)✓ β.2
runtime introspection(rails console + reflection)constantize 真實解析、AASM event、hash dispatch 求值高(load Rails app + 隔離)V4-γ 評估、本輪未做
ruby-lsp-rails(完整 LSP)所有 Rails 知識(routes / model / view)高(LSP 啟動 + 接 MCP)未做(LSP 啟動成本對 benchmark 太重)
本輪實驗 5 條件

把 V2-grep(純 LLM 自行搜尋)、V2-rdx(rubydex 工具箱可用)、V3-sel(prompt selector，用 prompt 引導 LLM 自己選工具)、V4-α(orchestrator，程式控制 routing，僅 rubydex+rg)、V4-β.2(orchestrator，加 rails-dsl + tokenizing scorer + hint vocab)五臂並排。前四個沿用 V4 報告數據,第五個本輪新跑。ORCH_ENABLE_RAILS_DSL=1 環境變數開啟 backend。Claude --strict-mcp-config / Codex --ignore-user-config 隔離（防止全域 MCP 工具污染對照組）。Value 9 題每題同一 judge 重判 V4-α + V4-β.2 兩臂（對齊評分嚴格度、附 V4-α 錨點驗 strictness drift）。

04

機器 6 題 aggregate ctx:5 臂並排

6 機器題的 context token 合計（in+cache+out，單位 K，代表 LLM 一輪問答消耗的 token 總量），五條件並排。看 β.2 列相對 α 的位移:Codex −30%(4024K→2825K)、Haiku −21%(4165K→3310K)、Opus +27%(1395K→1770K)。Codex 的 ctx 大降跟 machine 全綠是同一個現象 —— rails-dsl 給的 Rails-knowledge backend 直接命中、不用反覆換 query。Opus ctx 反向是因為它在 case-2/5/14 對 backend 結果再呼叫 rubydex 二次驗證。

β.2 並沒有解決「強模型 ctx 負擔」—— Codex 雖然從 ×9.7(α 對 V3sel)降到 ×7.5,仍遠高於 V3sel 基線。真實的故事是「Codex 的 ctx 砍法靠 rails-dsl 命中、Haiku 的 ctx 砍法靠 case-11 拒絕作答(用 0 換 ctx)」—— Haiku case-11 ctx 從 1209K 變 80K 看起來省、實際是 quality bad。把 ctx 跟 quality 同時看才對,單看 ctx 會 over-claim 效率提升。

對誰值得 + 該怎麼開

把 §05 收成 actionable:對哪個模型、哪些題型該開 rails-dsl,哪些不該。

模型machine 結論value 結論淨判斷 / 建議
Codex 強✅ recall +14pp 全綠 / ctx −30%❌ judge −2.5(計數 + silent bug 全崩)⚠️ 看任務目標決定 —— 純「找哪些檔」型(machine)開,「列舉 / 計數 / 推理 silent bug」型(value)別開

關鍵 takeaway

β.2 不是 universal 升級,是 model-dependent 而且強模型還是 machine ↔ value 二次 trade-off。Codex 上開 β.2 像在「強模型送 RoutingPower-up、但 reasoning 變懶」之間做選擇 —— machine 任務開、value 任務關。Opus 上 β.2 拿到 case-7 polymorphic 救起的個案勝、其他 wash。Haiku 上不該開。實務建議:按題型 + 模型 selectively enabled,而非無條件 on。

該怎麼 selectively 開(實作建議)

  • 勝場(對 Codex / Opus 開):Rails-aware locate 題 —— delegate 鏈、routes helper、polymorphic 關聯、ActiveRecord has_many/belongs_to 反查。case-1/11 / case-7 silent bug(對 Opus)是典型。
  • 敗場(對 Codex 關):窮舉計數題 —— case-4/4A/8 這類「列出所有 X 引用 + 總數」題。rails-dsl 給的「Rails-knowledge 命中」上下文讓 Codex 早收手,改回 V3-sel / V2-grep 拿全候選。
  • 敗場(對 Haiku 全關):Haiku 對 1-shot high-confidence 答案敏感(case-11 主場 bail),弱模型反而需要自己一步步搜出來的 ctx-rich 過程。直接用 V3-sel。
  • 長期:讓 LLM 在 prompt 自帶 capability hint(強/中/弱),orchestrator 按 hint 動態決定要不要 enable rails-dsl —— 跟 V4-α 結論一致(IA locate 也是 model-dependent),都該升級成 model-aware classifier。

5 個變化最大的個案中,挑兩個最能說明 β.2 「給太多 high-confidence 上下文反而傷」的:Haiku case-11 主場 catastrophic、Codex case-7 silent bug explain-away。

case-11 是「給 controller#action,反查所有 routes / URL helper / link_to 呼叫點」題 —— 純 Rails routes DSL 題,本就是 rails-dsl backend 該大勝的指定主場。看數字:ctx 從 1209K 暴砍到 80K、rdx 從 19 變 0、reads 從 8 變 0 —— backend 一次給回完整答案,Haiku 連 1 個 rubydex 呼叫都沒打、連 1 個檔都沒讀。理論上這是 β.2 教科書級別的勝利。

但 Haiku 接著做了一件出乎意料的事:它仍想自己 Read 檔案驗證,然後撞到工具的 cwd 跟 project_root 不一致(cwd 在 /Users/liu/ruby_indexer、project_root 在 /Users/liu/sunnyfounder/e_trading),回了「不能從 cwd access project_root」的 message,直接拒絕作答。

這是 β.2 設計缺陷的具體展現

rails-dsl 給 1-shot high-confidence 答案後,弱模型會被「project root 明確、但我的工具 cwd 不對」這種看似 instruction 衝突搞亂(它沒意識到 backend 已給了答案、自己再驗證會撞 cwd)。V4-α 的 1209K-ctx / 19 calls 答案 Haiku 自己一步步搜出來、沒有這個衝突 —— 過程更貴、但結果是 ok。

case-7 是「找 production code 裡 polymorphic source_type 字串裡的 silent bug —— 字串會被 classify.constantize（字串轉成 Ruby 常數名稱並嘗試取得該 class）解到頂層常數，但對應的真實 model 在 namespace 裡，於是 production runtime 會 RuntimeError」題。答案核心是 'income_expense' —— 它 constantize 到頂層 IncomeExpense（不存在），但有個 namespaced model ElectricityAccounting::ChargesCalculation::FixedCharge::Models::IncomeExpense 確實存在，只是不在頂層 → silent bug。

Codex 在 V4-α 給了 part(列了 Revenue::Due 等真 bug、漏 income_expense)。換 β.2 後,rails-dsl 餵的 polymorphic 關聯上下文裡顯式列出了 ElectricityAccounting::ChargesCalculation::FixedCharge::Models::IncomeExpense 這個 namespaced model 的存在,於是 Codex「合理化」掉:既然有這個 model、那 'income_expense' 對應的就是它(named with high confidence),不是 bug。

這是 rails-knowledge 反向

Codex 用 rails-dsl 給的 namespaced model 上下文把 silent bug explain away —— backend 給的不是錯資訊,而是「太多正確但 misleading 的相關上下文」,讓強模型在 high-confidence 下對「constantize 解的是頂層常數、不是任何 namespaced 同名類」這個 Rails 真實 runtime 行為失察。對 Opus(中模型)同樣的上下文反而救起來(Opus bad→ok),因為它走「A 類真 polymorphic vs B 類 sentinel」的分組推理路徑、明確指認 income_expense 是 silent bug。同 backend、同題、不同推理風格 → 反方向結論。

08

結論 + V4-γ go/no-go

① rails-dsl backend 真的「工作」:對 Rails routes、polymorphic 關聯這類 DSL 知識題,backend 一次命中 + ctx 砍法明顯 —— Codex case-1 delegate(+36pp / −45% ctx)、case-11 routes(Codex 1.0 / Opus 1.0)、Opus case-7 polymorphic(bad→ok)都是真實的勝場。不是「沒效果」,是「有效果但代價深」。

② 代價是 model-dependent + 題型相依的雙重 trade-off:同樣的 backend、同樣的高 confidence Rails 知識上下文,對 Codex 在 machine 是大勝、在 value 是大降(case-4/4A 計數 −60%、case-7 explain away);對 Opus 個案性勝(case-7);對 Haiku 雙降(主場 case-11 拒絕作答 + main path 受傷)。強模型還多一層 machine ↔ value 二次拉鋸,因為「Rails-knowledge 上下文」對 locate 任務是助攻、對 enumeration 與 silent-bug-reasoning 任務是干擾。

③ case-11 反諷揭露 β.2 真正的設計缺陷:不是 backend 知識不對,是「給太多 high-confidence 1-shot 上下文,弱模型反而被看似 instruction 衝突的訊號(project_root 在 X 但我 cwd 在 Y)搞亂,直接 bail」。V4-α 的「LLM 自己一步步搜」反而沒這個 surface。backend 越強、prompt-to-LLM 介面越要小心高 confidence 訊號跟 LLM 自我驗證之間的張力。

④ V4-β.2 真實的價值是 model + 題型雙重 routing,不是 backend 本身。光裝上 rails-dsl 並打開 ORCH_ENABLE_RAILS_DSL=1 不會 universal 升級;需要 classifier 按「模型強弱 + 題型 locate vs count vs reasoning」動態決定要不要 enable rails-dsl。跟 V4-α 結論「IA locate 是 model-dependent」是同一個 pattern 的延伸 —— 整個 V4 系列的 actionable 都是「外層 routing 需要 capability-aware」。

V4-γ runtime backend go/no-go

本輪也順帶釐清 V4-γ(runtime introspection backend,load Rails app 做 reflection)的可行性。case-7 silent bug 是 V4-γ 該大勝的指定主場(只有真實 runtime 知道 constantize 解到哪)—— 但 β.2 結果矛盾:Opus 升(rails-dsl polymorphic 上下文有用)、Codex 降(同樣上下文 explain away)。若 γ 給更強的「runtime 確認 income_expense 在頂層不存在」訊號,理論上 Codex 該被強訊號逼回正確答案;但 β.2 case-7 經驗顯示強模型遇到強 backend 訊號可能再 explain away(找個其他原因合理化)、不是直接服從。實作成本(Rails app 啟動 + 隔離)+ 不確定能不能解決 explain-away → V4-γ 仍 no-go,跟 α 報告結論一致。下一步該是 model-aware classifier(β.3?)而不是再多裝 backend。

如何看這些數字

每格只跑一次(n=1),單次變異很大。所有結論都是方向性訊號;單題的小差異(尤其 ±1 個 part/ok、或 case-10 Opus ok→part 這類 random drift)別當定論。要可發布的強主張需每格 N 次取中位。本報告對 β.2 的判斷「強模型雙刃劍 + Haiku 雙降」的方向訊號夠強(Codex value −2.5、Haiku case-11 catastrophic 都不是 ±1 邊界),但模型內個案的具體解讀(case-10 Opus 反向、Haiku case-8 救起)需要更多 N。

09

code line 追蹤:三輪實驗、兩個教訓

V4-β.2 結論「工具做到頭了」→ 改假設：把 L3 寫法（動態 constantize、interpolated i18n key）退役，讓 LLM 靜態看穿。跑了三輪（v4c / v4d / v4e），每輪都有具體發現。假設：改 code 讓 LLM 看穿比加工具更根本。n=1，看方向。

教訓 1：常數要放在模型搜尋路徑上，不是 view 層

三輪最關鍵的發現是搜尋層級的問題。LLM 在問 source_type 的 polymorphic 行為時，搜尋路徑是 domain 層（models / use_cases / entities），不是 view_model 或 context 層。v4c / v4d 的常數放在 view / context，LLM 的 find_code 呼叫根本不會路由到那裡——不是常數沒用，是模型永遠找不到它。

v4e 把 CUSTOM_SOURCE_TYPE_STRINGS 放在 models/entry.rb 緊鄰 belongs_to :source, polymorphic: true，加上「這些值 constantize 會 raise NameError」的注解。結果：Opus 和 Codex 都找到了，Opus 還用它推導出比 answer key 更深的 bug（income_expense.rb:29 的 has_many :records, as: 'source' 寫入 FQN 與短字串並存導致的衝突）。

教訓 2：case-7 的本質是設計歧義，不只是 code 不透明

即使 Opus 拿到常數後給出了最好的分析，income_expense 仍然很難被強制歸類為「silent bug」——因為技術上它有對應物件（透過 belongs_to :income_expense FK），只有從不被呼叫的 polymorphic path（entry.source）才會炸。這是潛伏 bug，不是活躍 bug，模型做出「有對應」的判斷其實有一定道理。Haiku 的問題更根本：ctx=1522K、max_turns 爆掉，換 code 解決不了搜尋效率問題。

✓ code line 有效的情境

  • 常數放在 domain model 層，緊鄰觸發 bug 的宣告（如 belongs_to :source, polymorphic: true）
  • 中強模型（Opus）能利用常數做出更深的分析，即使 grade 不變也提升了推理品質
  • case-10 類型（直接 i18n key 引用鏈）：常數讓模型精確計算「要動 4 個 file」，Opus 穩定 ok

✗ code line 無效的情境

  • 常數放在 view_model / context 層 → 模型搜尋路徑不到那裡
  • Haiku 類型弱模型：搜尋效率問題，常數能不能找到是次要，45 turns 就 max_turns 爆了
  • 設計本身歧義（FK bypass + polymorphic latent bug）：code 再清楚模型也可能合理化「有對應」
三輪總結

「改 code 讓 LLM 看穿」假設在 domain model 層有效，在 view / context 層無效。v4e 的 Opus 結果是迄今最清楚的正向信號：找到常數 → 自動找到更深的 bug 。但對 Haiku 弱模型和設計本身歧義的 case，code 透明度已不是瓶頸。

這個假設的最佳應用場景是：把現有 L3 anti-pattern（特別是 polymorphic bypass + 自訂短字串 combo）的 domain model 加上顯式常數 + 注解，讓中強模型（Opus/Codex 等級）能靜態得出正確答案，而不需要多輪 ripgrep 探索。
