把 routing 拉到 LLM 外面:
外層 orchestrator MCP,真的比叫模型自己選好嗎?

V3 證明 prompt 層的「叫模型自己當選擇器」路由不可靠(結構主場不叫 rubydex、字串盲區猛叫)。V4 把 routing 拉出 prompt、做成外層 MCP:LLM 只看到一個 find_code(query, hints?) 工具,server 內部用 rule-based intent classifier 路由到 rubydex(結構)或 ripgrep(字串),再加一個 Aider 風格的 get_codebase_map primer。這次把 V4-orchestrator 跑滿全 15 題 × 3 模型,跟 V2-grep / V2-rubydex / V3-selector 放在同一把尺下比。問題:外層主動路由,到底有沒有兌現 rubydex 的價值?

一句話結論

外層 orchestrator 不是萬靈丹:它的價值強烈題型相依。「找位置 / 找全少量引用」題持平甚至更好(弱模型還省 context),但「列舉 / 計數所有引用」題被 find_code 的 top_k 截斷打崩,盲區推理題照樣無解。

機器 6 題(file-level recall)V4 持平/略升:Haiku 平均 .94→.96、case-1(delegate 鏈)更從 V3 的 0.64 衝到 1.0 —— find_constant_references enrichment 一次抓全引用。value 9 題裡的 計數窮舉題(case-4/4A/8)一度被 top_k 截斷打崩:原始 V4-α 只回 top_k=10 候選、不回 total → LLM 把「~100 筆引用」報成 33-38,三模型 value 從 V3sel 的 7.5/8.0/8.5 掉到 5.0/6.0/5.5。但這是可修的工程債:讓 find_code 回確切 total + 不截斷引用集後重跑,計數題 9 格救回 8 ok —— 強模型 Opus/Codex value 追平 V3sel(8.0/8.5)、弱模型 Haiku 回到 6.5。只剩 case-4A(無 FQN、要自己找全 4 類別並加總)haiku 仍漏:orchestrator 給對了 per-class total,卡在 haiku 不會整合(opus/codex 則整合成功)。機器 6 題(算「哪些檔」、top_k 夠)本就沒崩,V4 持平/略升(case-1 delegate 0.64→1.0)。

效率則是模型相依的反轉:弱模型 Haiku 用 find_code 取代廣搜 → 6 機器題 ctx 全省(−14%~−50%);但強模型 Opus 本來就惜字如金(V3 幾乎不主動叫工具),被 orchestrator + primer 強灌候選後 ctx 反增 +120%。呼應 V3 的「MCP 載入稅對 Opus 是淨負擔」。

三個 caveat:① case-7 silent bug 三模型 9 格仍全摃龜(orchestrator 對推理盲點無增益)。② V4 toolset = V3 + orchestrator 疊加;曾試 --disallowedTools 拔掉 raw grep/Read 強制單一通道,recall 反而更崩(case-2 1.0→0.0)—— pure single-channel 不可行,hybrid 才是現實。③ 全程 n=1。

01

V2 → V3 → V4:從「丟工具箱」到「外層 orchestrator」

四代測的是「該怎麼把一個程式碼索引接進 LLM」的不同姿勢。任務、題庫不變,只變「rubydex 怎麼接」—— 從 prompt 點名(V1)、丟工具箱(V2)、叫模型自己路由(V3),到 V4 把路由拉到 LLM 外面變成 MCP server。

版本對 rubydex 的姿勢對照組可比性
V1強迫:「請用 rubydex」 vs 「請用 grep」(prompt 點名工具)with / without不同 prompt 格式 + 自己一套 answer key + 無 case-14 → 只能當前身參考,不上同一把尺
V2放工具箱:prompt 模糊、工具全開,載不載 MCP 由環境決定,模型自選用不用grep-only / rubydex-available乾淨可比(V2 已修 MCP 污染)
V3選擇器:rubydex 在工具箱 + 策略明寫「先廣 grep、結構引用才用 rubydex 驗證、字串/動態盲區走 grep」 → 模型自己按題型路由selector(單一 arm)前作;證明 prompt-selector 不可靠
V4外層 orchestrator:routing 拉出 prompt、做成 MCP server。LLM 只看到一個 find_code(query, hints?),server 內 rule-based intent classifier 路由 structural→rubydex / string→rg、0 命中退 rg,另加 get_codebase_map(Aider 風格 repomap primer)。不靠模型自律orchestrator(單一 arm)本報告主角
本報告的 4 條件

主表把 V2-grep(純 LLM 基線)、V2-rubydex(光放工具箱)、V3-selector(叫模型自己路由)、V4-orchestrator(外層主動路由)四條件擺一起。前三個沿用 V3 報告的乾淨數據,第四個本輪新跑。機器題 file-recall 自動算;value 題每題同一個 judge 重判 V3sel + V4orch 兩臂(對齊 strictness、附 V3 錨點驗 drift)。V4 是 hybrid:orchestrator 工具 + LLM 仍可自選 raw grep/Read。

04

find_code 取代了 grep:沒人再用 rg

V3 的問題是「叫模型自己挑」—— selector 實際呼叫 rubydex 又稀疏又沒對準題型。V4 把這個決定拿走了:LLM 只有一個 find_code 通道,要搜就只能叫它。下表是每題 find_code 呼叫次數(V4orch arm)—— 對比 V3 selector 的稀疏亂叫,V4 是每題穩定使用。

case型Haiku find_codeOpus find_codeCodex find_code
find_code 取代了 broad search

三模型每題都穩定叫 find_code(從 2-3 次到計數題的數十次),而且 沒人再用 grep / rg —— 看 transcript,模型的「繞道」幾乎都是 Read 去確認候選的完整 context,不是繞過 orchestrator 自己廣搜。find_code 確實接管了 broad search 的角色。計數題(case-4/4A/8)find_code 次數特別高(Codex case-1 叫 44 次、case-2 叫 51 次)是模型在反覆換 query 想湊齊被 top_k 截掉的引用 —— 但 top_k 截斷不是換 query 能救的,所以叫再多次也補不回。

對照 V3:selector 是「拜託模型自己挑、結果結構題不叫盲區題猛叫」。V4 把 routing 從 prompt 搬到 server,routing 行為本身可靠了(intent classifier 穩定路由)。但這只解決了「叫不叫對工具」,沒解決「工具回傳完不完整」—— 後者(top_k 截斷)才是 V4-α 計數題崩的真兇,留給 V4-β。

05

讀檔 / token:弱模型省、強模型反增

6 機器題的 aggregate context token(in+cache+out,K),四條件並排。基線要看清楚:V4 的省是 相對 V3-selector,不是相對純 grep。Haiku 的 orchestrator 條比 selector(最高)明顯矮、跟 rubydex 持平,但仍比純 grep baseline 高(載 MCP+primer 有底稅);Opus 的 orchestrator 條則是全場最長。V4 context 是模型相依的反轉。

弱模型 Haiku 全省:V3sel→V4orch 六題 ctx 全降 —— case-6 340K→172K(−49%)、case-14 223K→112K(−50%)、case-1 611K→436K(−29%)。find_code 把廣搜的探路檔免掉。但強模型 Opus 全增:case-5 81K→212K(+162%)、case-6 76K→179K(+136%)、case-13 182K→405K(+123%)。Opus V3 本來惜字如金(幾乎不主動叫工具、ctx 極低),V4 強制走 find_code(每次帶候選進 context)+ 開頭 get_codebase_map primer → context 反被灌大。呼應 V3 的「MCP 載入稅對 Opus 是淨負擔」:orchestrator 對已經高效的模型不划算。

三層地雷分類:哪些 Ruby pattern 對 AI 爬 code 友善 / 是地雷

把 15 題按「AI 靜態爬 code 的瓶頸在哪」分三層。判準是 靜態字面 vs 動態求值:答案顯式寫在 code 裡 → 工具撈得到(L1);要 LLM 多步推理整合 → 卡模型能力(L2);要 Ruby 求值才確定 → 靜態分析的黑洞(L3)。主表每題左欄已標 L1 / L2 / L3。

層Ruby patterncase瓶頸 / 為什麼
L1 靜態顯式FQN 引用 / has_one+delegate / Rails routes / callback symbol / namespace 計數 / YAML 固定字串1·14·11·4·8·9·5·6答案顯式在 AST 或字面字串裡,rubydex enrichment 或 grep 直接撈。AI 友善。(計數題 case-4/8 修 top_k 後才穩)
L2 半動態 / 需整合多 namespace 同名 class 計數 / 字串組方法名 / class_eval / method_missing4A·2·3·12資料靜態可得,但要 LLM 多步推理(篩選+加總、推斷生成/轉發目標)。強模型行、弱模型卡(case-4A haiku 47 vs opus 141)。
L3 runtime 求值polymorphic constantize / hash dispatch table / 動態 i18n key7·13·10答案要 執行 Ruby 求值才確定(constantize 解到哪、hash[key] 映射、#{var} 拼出的 key)。靜態分析(rubydex/grep/AI 讀 code)本質摸不到。AI 地雷,只有 runtime introspection(V4-γ)摸得到。
關鍵判準:靜態字面 vs 動態求值

L1 與 L3 的分水嶺最值得記:同樣是「字串指 class」,case-5(Sidekiq YAML worker)是 L1 友善、case-7('income_expense'.classify.constantize)是 L3 地雷。差別不在「有沒有用字串」,在那個字串需不需要 Ruby 求值才知道指向誰 —— YAML 字面固定(grep 撈),classify.constantize 要執行(且可能解到不存在的頂層常數 = silent bug)。Ruby 的 constantize / 手動 dispatch table / #{} 拼 key 對任何靜態分析都是黑洞。

✅ AI 友善的 Ruby pattern

  • 顯式 FQN / namespace 常數引用
  • association(has_one/belongs_to)+ delegate 宣告
  • Rails routes / callback symbol(symbol→def 可追)
  • 固定字面字串指 class(YAML config、class_name: '...')

❌ AI 地雷的 Ruby pattern

  • str.classify.constantize / polymorphic 解析(case-7)
  • 手動 hash dispatch table { 'k'=>Class }[key](case-13)
  • #{var} 動態拼字串 key / 方法名(case-10、case-2)
  • (半地雷)class_eval / method_missing 動態生成 — 強模型可推、弱模型難

給想評估自家 codebase「AI 友善度」的人:L1 pattern 放心交給 AI;L2 給強模型 + 確保工具回完整資料(別截斷);L3 pattern(尤其 constantize 那類)是 AI 維護黑洞 —— 考慮重構成顯式映射,或別期待靜態 AI 工具找全。這也呼應 V4 主結論:外層 orchestrator 解決 L1(+ 修 top_k 後的計數)、幫不到 L3。

勘誤與方法

這輪查證時抓到的問題,寫出來才誠實。

① answer key 修正:case-12 ledger 型別 1 → 4

舊 key 寫 ledger_inferred_types=1。但讀 ledger_factory.rb 的 live code:LedgerFactory#steps 回的是 Success(user_ledger) = build_ledger_by_user(...) 的結果,依「是否換約 × 是否含稅」回 4 種型別(2 UserLedger + 2 Decorator);key 寫的那個 Ledger 只是傳進去的內層參數,從不直接回傳。→ 正解是 4,答 4 的模型是對的;key 已修。

② case-7 確認 key 沒錯:income_expense 真的是 silent bug

有人(模型)說 income_expense 對到 …FixedCharge::Models::IncomeExpense 所以不是 bug。但它存進 Rails polymorphic 關聯,讀 .source 時 Rails 會 constantize 頂層 IncomeExpense(不存在)→ 仍是 silent bug。namespaced 那個 model 構不到。→ key 正確,是模型判錯。

③ answer key 修正:case-4 / case-4A 「扣 class def」是 bug(2026-05-26 全 key 重驗)

為了把整批 key 釘乾淨,把剩下 7 個沒驗過的 key 對 live code 重驗了一輪。發現 case-4 / case-4A 的計算公式裡有個共同 bug:扣掉了「class def 自身」那一行,但實際上 class Invoice < ApplicationRecord 那行不含 FQN 字面、不在 raw count 內,本來就不該扣。修正後:

  • case-4:total 101 → 102(+1)。原本答 102/106 的 LLM 其實更接近正解;判讀無變化。
  • case-4A:per-class +1,total 141 → 145(+4)。Opus/Codex 的純 grep 結果(150/151)從「過報 ~9」變「過報 ~5-6」→ recall-first 下從 part 升到 ok。**這讓 Codex 的 grep≤rubydex 變成平手(8.0=8.0),不是嚴格 < —— 不過 selector 仍勝**。
  • 其他 5 個 key(case-3/8/9/10/11)逐項對 live code,全部 ✅ matches,沒改。
方法:為什麼要「per-case 重判」

第一輪我為了平行,每個模型派一個 judge,結果不同 judge 對同一份 key 套出不同嚴格度(case-10:某 judge 全 bad、另一個全 ok)→ 跨模型分數被污染。改成每題一個 judge 判該題全 9 格後,同題內跨模型/跨條件才真的可比。本報告主表用的是重判後的乾淨資料。

07

結論

① 外層 routing 本身可行:rule-based intent classifier + 單一 find_code 工具真的把路由從 prompt 拉出來了 —— 三模型在結構題乖乖走 rubydex、字串題走 rg,不再像 V3 selector 那樣亂挑。case-1(delegate 鏈)從 V3 的 0.64 衝到 1.0 是最好的證據。

② 計數窮舉題的崩是可修工程債,不是 orchestrator 本質限制:原始 V4-α 的 top_k 截斷讓 case-4/4A/8 報殘缺數字(三模型 value 掉到 5.0/6.0/5.5);診斷後讓 find_code 回確切 total + 不截斷引用集,重跑救回 9 格中 8 ok,強模型 Opus/Codex value 追平 V3sel(8.0/8.5)。剩 case-4A haiku 仍 bad —— 那是「多目標 + 需語意篩選哪些 class 算 + 加總」卡 LLM 整合(opus/codex 整合成功),不是工具給不出資料。

③ 效率是模型相依的反轉:弱模型(Haiku)find_code 取代廣搜 → ctx 省 33%、recall 還升;強模型(Opus)本來惜字如金,被 orchestrator + primer 強灌候選 → ctx 反增 120%。orchestrator 對已經高效的模型是淨負擔。

④ pure single-channel 不可行:試過 --disallowedTools 拔掉 raw grep/Read 強制只走 find_code,recall 反而更崩(case-2 1.0→0.0、LLM 在部分題直接認輸)。實務上 hybrid(orchestrator + LLM 自選 raw 補強)才是現實 —— V4 是加在工具箱、不是取代。

V4-β go / no-go

值得做。本輪已修掉最大硬傷 top_k 截斷(find_code 回確切 total + 不截斷引用集),計數題 9 格救回 8 ok、強模型 value 追平 V3sel。剩兩個課題:(1) case-4A 多目標整合 —— orchestrator 給對 per-class total,但弱模型(haiku)不會「語意篩選哪些 class 算 + 加總」(opus/codex 會),這是 LLM 能力邊界、非工具能全補。(2) 強模型負擔 —— orchestrator 對 opus 是淨 context 增,別無差別強推。β 方向(ruby-lsp-rails 補 route/association、rails runtime 補 constantize/polymorphic)仍合理。

如何看這些數字

每格只跑一次(n=1),單次變異很大。所有結論都是方向性訊號;單題的小差異(尤其 ±1 個 part/ok)別當定論。要可發布的強主張需每格 N 次取中位。
