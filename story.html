Ruby Code Search × LLM · V1→V5 完整故事

能不能用工具讓 LLM
更會查 Ruby code？

又準、又快、又省。我跑了五輪 benchmark 想回答這個問題。
最後的答案有點出乎意料：堆工具有極限，真正的根本解是改幾行 code。

完整故事 V1 V2 V3 V4-α V4-β.2

主旨

一個問題，貫穿五輪

LLM 幫你改 code 前，得先看懂 code —— 找出某個 method 被誰呼叫、某個關聯指向哪、改個名字會動到哪些檔。Ruby 又特別愛用「動態」寫法，很多東西不寫在明面上。

所以問題很單純：給 LLM 一些程式碼搜尋工具，它會不會查得更準、更省 token、更便宜？

宣告意圖

怎麼測

做了一個 benchmark：15 題真實 Rails codebase 的難題（找關聯、數引用、追動態方法、揪 silent bug），每題都有人工驗證過的標準答案。每題用 3 個模型（一弱一中一強）跑，比較「給工具 vs 不給工具」「哪種題配哪種工具」。

衡量三件事：準（有沒有找到正確答案）、省（消耗多少 token）、穩（換模型結論會不會翻盤）。

怎麼讀分數

後面出現的分數是 0.0（全錯）到 1.0（全對）。工具分兩種：grep（純文字搜尋，LLM 內建）、靜態索引（先把 Ruby 結構建成索引再查，像 rubydex / RubyLSP 那類）。

證明過程

五輪，每輪學到一件事

過程中試了很多搜尋策略跟驗證方法，這裡只留每輪真正的結論。

  • V1 – V2
    靜態索引不是萬靈丹
    對「結構題」（找關聯、找 caller）靜態索引贏 grep；但對「字串題」（YAML 設定、字面字串）它反而是負擔——純 grep 一樣準還更快。工具有適用範圍。
  • V3
    叫模型自己選工具 → 不可靠
    在 prompt 裡寫「結構題才用索引」，賭模型自律。結果：弱模型幾乎不用、強模型在不該用的題猛用。選工具的判斷不能交給模型。
  • V4
    把選工具搬到程式層 → 找檔變準變省，但模型開始想錯
    改由外層程式（orchestrator）按題型自動派工具。找檔大幅變準、token 砍三成。但出現反效果：工具一次餵太多「高信心」上下文，強模型會把真 bug 合理化掉、弱模型會被搞亂到拒答。工具給的是資料，不是判斷力。
  • V5
    兩個方向：拆分工沒用，改 code 直接見效
    試 A「便宜模型找檔 + 強模型推理」：推理那段 token 砍到極低，但總品質掉了——因為便宜模型找檔本來就比強模型差，給的料不全。
    試 B「改寫 code」：把一個動態生成方法的寫法改成普通寫法後，模型立刻精準命中。這是整個系列第一次看到「改 code」直接見效。

核心發現

哪種題，該用哪種工具

把 15 題歸類後，規律很清楚——工具有沒有用，看題型：

題型例子工具效果怎麼做
結構關係 找關聯、找 caller、改名影響面 索引大勝 用靜態索引，grep 會漏
命名 / 計數 同名 class 散在多個 namespace 索引大勝 用索引（grep 0.67 → 索引 1.00）
字串 / 設定 YAML 指 class、monkey-patch grep 就夠 grep 滿分，加索引是浪費
動態方法 用迴圈/eval 生出來的方法 工具都沒用 改 code 或硬靠模型推理
設計歧義 polymorphic 存自訂短字串、跑起來才知道對不對 全軍覆沒 只能改 code（見下）

上面兩列綠的，是工具的主場；中間 grep 就夠；下面兩列紅的，再強的工具、再強的模型都解不漂亮——因為答案根本不寫在靜態結構裡。這就帶到最後的結論。

結果總結

三個結論

一句話

工具能幫 LLM「找對地方」，但幫不了「想清楚」。最划算的投資不是堆更多工具，是改掉少數讓靜態分析看不穿的 code 寫法——工具會隨模型改版過時，好 code 不會。

  1. 根本解是改 code，不是堆工具。動態生成方法、字串拼 class 名這類寫法，把它改成普通寫法後，連最弱的模型都能一眼看穿。而且這個改善對 grep、LSP、所有工具、所有模型都有效——不像工具，下個模型發布就可能要重調。
  2. 工具要按題型用，不要一招打天下。結構題用靜態索引、字串題用 grep。混用只會白付成本，甚至（V4 證明）餵太多上下文還會害模型想錯。
  3. 有一種寫法是真正的地雷。「用字串在執行期才拼出名稱、又沒有固定前綴可搜、還要跨好幾個檔才能反推對應」——這種三重組合是整個 benchmark 唯一全軍覆沒的，所有工具分數都低於 0.5。這類該優先退役。

給想套用的人：三層準則

如果你也想檢視自己的 Ruby codebase，照這個優先序：

層級寫法怎麼辦
盡量避免 字串拼 class 名（"#{x}".constantize）、polymorphic 存自訂短字串、用 class_eval/變數名 define_method 動態生方法 改成顯式寫法：registry hash、寫死的 def、存完整 class 名
換工具 多 namespace 同名 class 別為了好搜而放棄 namespace（那是好設計）。改用靜態索引工具，它認得
別動它 method_missing 轉發、public_send("固定前綴_#{var}")、monkey-patch、YAML 字串 數據證明 grep/LLM 都能處理。退役它們是浪費工

最反直覺的一點

我們本來以為「名字是變數的動態寫法都該退役」。數據打臉：public_send("do_#{state}!")、method_missing 這些看起來很動態的，反而都拿接近滿分——因為有固定前綴可搜、或範圍很局部。真正的地雷不是「動不動態」，是「能不能被靜態追蹤」。
