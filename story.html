Ruby Code Search × LLM Benchmark · V1→V5

工具能不能讓 LLM 更會查 Ruby code？

目標只有兩個：查得更快、花得更少——前提是答案品質不能掉。15 個真實 Rails case、3 個模型、三種搜尋策略，跑了五輪，本頁說明測了什麼、怎麼測、哪個數字得出哪個結論。

結論先講

三句話總結

如果只看這段

① 工具有沒有用，取決於題型：結構關係題（找 caller / 關聯穿透 / namespace 解析）靜態索引明顯贏 grep；純字串題 grep 就到頂，加索引是浪費。
② 「該用哪個工具」的決策要放在程式層、不能交給模型自己判斷——但即使派對工具，餵太多上下文反而會讓強模型把 bug 合理化掉。
③ 有一類寫法（執行期用字串建構名稱、且無固定前綴、又要跨檔反推）靜態工具全軍覆沒，再強的模型也救不回——這類唯一的根本解是改 code，且效益不隨模型版本過時。

以下是怎麼測出這三點的：方法、15 個 case 的成績、五輪的決策過程、被挑戰時站不站得住。

背景

為什麼測這個

LLM 要改一段 code，前提是先看懂它：這個 method 被誰呼叫、這個關聯指到哪個 model、改個名字會波及哪些檔。Ruby / Rails 大量使用「執行期才成形」的寫法（delegate 隱式 method、method_missing 轉發、constantize 字串轉常數、polymorphic association），很多對應關係不寫在原始碼明面上，grep 抓不到、靜態分析也未必追得到。

問題具體化成一句：給 LLM 程式碼搜尋工具，它能不能查得更快、更省 token，又不犧牲答案品質？更快跟更省其實同源——少繞幾輪、少讀幾個檔，token 跟時間一起降。

3 模型、兩種評分、三種搜尋策略

  • 模型：一弱（Haiku 4.5）、一中（Opus 4.7）、一強（GPT-5.5 / Codex）。每格只跑一次（n=1）。
  • 評分：6 個「找檔題」用 file-recall（標準答案是檔案清單，算涵蓋率 0–1，與版面格式無關）；9 個「判讀題」由單一 judge 評 ok / part / bad（換算 1 / 0.5 / 0），固定同一個評分者避免標準漂移。
  • 搜尋策略：純 grep（LLM 內建文字搜尋）、靜態索引（先把 Ruby 結構建索引再查，類似 rubydex / RubyLSP）、orchestrator（外層程式按題型自動派工具，模型不自己選）。

過程

五輪實驗：每輪的結果與決定

每輪都是被上一輪的結果逼出來的。下面標清楚「測到什麼 → 所以決定做什麼」。

V1–V2：給工具有沒有用？

結果：對照同一批 case 的 grep vs 靜態索引——結構題（case-1 關聯穿透 .62→.81、case-4 namespace .67→1.0）索引明顯贏；字串題（case-5 YAML、case-6 monkey-patch）兩者都 1.0，加索引純屬多餘。工具有適用範圍，不是越多越好。

決定既然不是全有用，真正要解的不是「要不要給工具」，而是「哪種題該用哪個工具」。下一步：試著讓模型自己判斷該用哪個 → V3。

V3：讓模型自己選工具

結果：在 prompt 寫規則「結構題才用索引、字串題用 grep」，賭模型照做。實測弱模型該用時不用、強模型不該用時猛用，命中率跟「工具全開、不分類」差不多——等於沒分類。

決定選工具的判斷靠模型自律不可靠。把這個決策從 prompt 抽出來，移到一層外部程式按題型硬性派工具 → V4。

V4：把選工具搬到程式層（orchestrator）

結果（好的一面）：外層程式按題型自動派工具，找檔變準也變省——case-1 強模型 recall 0.64→1.0、整體 token 砍約三成、case-11 routes 從 .83 升到 1.0。方向對了。

結果（壞的一面）：工具一次餵大量「高信心」上下文後，模型反而想錯。case-7 強模型把真 silent bug 合理化掉（part→bad）——看到一個同名但在別 namespace 的 model 存在，就認定字串有對應，忽略 constantize 實際解不到。同一份上下文，中模型卻靠分組推理答對。工具給的是資料，不是判斷力。

決定兩條線索：(1) 強弱模型在「找檔」和「推理」上表現分裂 → 想把兩件事拆給不同模型做；(2) case-7 / case-3 這種題答案根本不在靜態結構裡，工具再強也沒用 → 該回頭改 code。V5 兩個方向一起試。

V5：兩階段分工，以及改 code

方向 A 結果（拆分工）：弱模型找檔、強模型只推理。推理那段 token 砍到極低（15 題合計約 175K），但整體品質掉了——找檔 recall 0.89→0.74、判讀 8.5→6.5（滿分 9）。瓶頸是弱模型找檔本來就差，給下游的料不全，再強的推理也補不回來。

方向 B 結果（改 code）：把 case-3 的 class_eval 動態生方法改寫成 6 個普通 def，同流程重跑，模型從「答得出機制但給不出行號」（part）變成「直接命中 def info 在第 17 行」（ok）。整個系列第一次看到改 code 直接把分數拉起來。

決定拆分工不划算（弱模型找檔是死結）；改 code 是唯一對「靜態無解」題有效、又不會隨下個模型發布而過時的根本解。收斂成一份「哪些寫法該退役」的準則（見下方結論）。

分析

為什麼 case-7 全軍覆沒

case-7 是唯一一題所有策略、所有模型都低於 0.5 的（grep .00 / 索引 .17 / orchestrator .33）。它的寫法是 polymorphic association 的 source_type 欄位不存 class 全名，存自訂短字串：

# source_type 存的是 'adjustment' 這種短字串，不是 class 名
belongs_to :source, polymorphic: true
# 'adjustment'.classify.constantize → Adjustment（頂層不存在）→ 解不到
# 真正的 model 是 Billing::Charges::Models::Adjustment，在 namespace 裡

三個原因疊加，才讓它無解：

  • 字串沒有固定前綴可錨定 → grep 無從下手（對比 case-2 的 do_ 前綴就好搜）
  • 對應的 class 在執行期 constantize 會失敗 → 靜態索引追不到這條斷掉的鏈
  • 要跨好幾個檔讀 dispatch 邏輯才能反推哪個字串對哪個 model → 超出單純搜尋

對照 case-3（class_eval）：它 grep 跟索引都是 .67，代表靜態索引對動態生成的方法完全沒幫助——method 不在語法樹裡，索引看不到。這兩題的共通點是答案根本不寫在靜態結構中，所以堆工具沒用，只能改 code。

結論

結論

三個，都對得回上面的數據：

  • 工具效果取決於題型，不是越多越好。結構題（case-1/4/13）靜態索引贏 grep；字串題（case-5/6）grep 就滿分，加索引浪費。混用甚至有害（V4 餵太多上下文 → case-7 想錯）。
  • 選工具的決策要放程式層，不是 prompt。V3 賭模型自律失敗；V4 程式按題型派工具才把找檔分數穩定拉起來。
  • 少數寫法靜態無解，改 code 才是根本解。case-7、case-3 這類「答案不在靜態結構裡」的，所有工具所有模型都低分；改寫後（case-3 已實測）立刻見效。而且改 code 的好處對 grep、索引、所有模型通用，不會因為下個模型發布而過時。

哪些寫法該優先退役

給想套用到自己 codebase 的人，按 benchmark 分數分三層：

優先序寫法證據改法

一個反直覺的點

「名字是變數的動態寫法都該退役」是錯的。public_send("do_#{state}!")（case-2）、method_missing（case-12）名字都動態，卻都接近滿分——因為有固定前綴可搜、或範圍局部。真正的弱項不是「動不動態」，是「能不能被靜態追蹤」：字串建構名稱 + 無固定前綴 + 跨檔才能反推，三者同時成立才無解。

限制

可能被挑戰的點，先講在前面

這份東西禁不禁得起追問，取決於下面這幾條有沒有講清楚。

  • 「n=1，憑什麼下結論？」——每格只跑一次，單次變異大。所以本報告只主張方向訊號，不主張精確值；像 case-7 全 <.5、case-4 索引 .67→1.0 這種大落差才當數，±0.5 的個案差異不解讀。要發表級結論需每格取多次中位數。
  • 「judge 是你自己評的，會不會偏？」——判讀題由單一評分者用固定 rubric（漏報重罰、過報輕罰）評，跨版本重判時附錨點驗標準漂移。但仍是主觀，沒有跑跨評分者一致性。
  • 「codebase 大小不同，數字能比嗎？」——所有條件（grep / 索引 / orchestrator）都在同一批 5,501 檔的專案上跑同一題，是同尺度對照，差異來自策略不是規模。但反過來說，結論只保證在「數千檔量級的生產 Rails」成立；小專案上 grep 的雜訊少，索引的優勢可能縮小。
  • 「case 是你挑的，會不會專挑工具贏/輸的？」——15 個 case 是先列 Ruby/Rails 寫法分類、再每類取題（見方法表），不是先看結果再選題。但 pattern 的分布權重（哪種寫法多）來自這組專案，未必代表所有 codebase。
  • 「只測 3 個模型、會不會換一個就翻盤？」——刻意取一弱一中一強。三模型分裂（同題不同結論，如 case-7）本身就是結論之一，但模型內個案仍需更多 N 才能歸納「某模型適合某策略」。
