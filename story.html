Ruby Code Search × LLM Benchmark · V1→V5

工具能不能讓 LLM 更會查 Ruby code？

15 個真實 Rails case、3 個模型、三種搜尋策略，跑了五輪。本頁說明測了什麼、怎麼測、哪個數字得出哪個結論。

總覽 V1 V2 V3 V4-α V4-β.2 數據

為什麼測這個

LLM 要改一段 code，前提是先看懂它：這個 method 被誰呼叫、這個關聯指到哪個 model、改個名字會波及哪些檔。Ruby / Rails 大量使用「執行期才成形」的寫法（delegate、method_missing、constantize、polymorphic association），很多對應關係不寫在明面上，grep 找不到、靜態分析也未必追得到。

問題具體化成一句：給 LLM 程式碼搜尋工具，它查得會不會更準、更省 token？哪種題該配哪種工具？換模型結論會不會翻盤？

3 模型、兩種評分、三種搜尋策略

  • 模型：一弱（Haiku 4.5）、一中（Opus 4.7）、一強（GPT-5.5 / Codex）。每格只跑一次（n=1）。
  • 評分：6 個「找檔題」用 file-recall（標準答案是檔案清單，算涵蓋率 0–1，與版面格式無關）；9 個「判讀題」由單一 judge 評 ok / part / bad（換算 1 / 0.5 / 0），固定同一個評分者避免標準漂移。
  • 搜尋策略：純 grep（LLM 內建文字搜尋）、靜態索引（先把 Ruby 結構建索引再查，類似 rubydex / RubyLSP）、orchestrator（外層程式按題型自動派工具，模型不自己選）。

五輪實驗，逐輪的問題與數據

V1–V2：給工具有沒有用？看題型

對照同一批 case 的 grep vs 靜態索引：結構題（case-1 找關聯穿透 .62→.81、case-4 namespace .67→1.0）索引明顯贏；字串題（case-5 YAML、case-6 monkey-patch）兩者都 1.0，加索引純屬多餘。工具有適用範圍，不是越多越好。

V3：讓模型自己選工具

在 prompt 寫規則「結構題才用索引、字串題用 grep」，賭模型照做。實測：弱模型該用時不用、強模型不該用時猛用，命中率跟「工具全開」差不多。選工具的判斷交給模型 prompt 不可靠。

V4：把選工具搬到程式層

改由外層程式（orchestrator）按題型自動派工具。找檔變準也變省——case-1 強模型 recall 0.64→1.0、整體 token 砍約三成；case-11 routes 從 .83 升到 1.0。

但出現反效果：工具一次餵大量「高信心」上下文後，模型反而想錯。case-7 強模型把真 silent bug「合理化」掉（part→bad）——它看到一個同名但在別的 namespace 的 model 存在，就認定字串有對應，忽略了 constantize 實際上解不到。同一份上下文，中模型卻靠分組推理答對。工具給的是資料，不是判斷力；資料給太多，強模型會用它說服自己。

V5：兩個方向

方向 A，拆分工：弱模型負責找檔、強模型只負責推理。推理那段 token 砍到極低（15 題合計約 175K），但整體品質下滑——找檔 recall 0.89→0.74、判讀 8.5→6.5（滿分 9）。原因是弱模型找檔本來就比強模型差，交給下游的料不全，再強的推理也補不回來。

方向 B，改 code：把 case-3 的 class_eval 動態生方法，改寫成 6 個普通 def。同一套流程重跑，模型從「答得出機制但給不出精確行號」（part）變成「直接命中 def info 在第 17 行」（ok）。這是整個系列第一次看到改 code 直接把分數拉起來。

為什麼 case-7 全軍覆沒

case-7 是唯一一題所有策略、所有模型都低於 0.5 的（grep .00 / 索引 .17 / orchestrator .33）。它的寫法是 polymorphic association 的 source_type 欄位不存 class 全名，存自訂短字串：

# source_type 存的是 'adjustment' 這種短字串，不是 class 名
belongs_to :source, polymorphic: true
# 'adjustment'.classify.constantize → Adjustment（頂層不存在）→ 解不到
# 真正的 model 是 Billing::Charges::Models::Adjustment，在 namespace 裡

三個原因疊加，才讓它無解：

  • 字串沒有固定前綴可錨定 → grep 無從下手（對比 case-2 的 do_ 前綴就好搜）
  • 對應的 class 在執行期 constantize 會失敗 → 靜態索引追不到這條斷掉的鏈
  • 要跨好幾個檔讀 dispatch 邏輯才能反推哪個字串對哪個 model → 超出單純搜尋

對照 case-3（class_eval）：它 grep 跟索引都是 .67，代表靜態索引對動態生成的方法完全沒幫助——method 不在語法樹裡，索引看不到。這兩題的共通點是答案根本不寫在靜態結構中，所以堆工具沒用，只能改 code。

結論

三個，都對得回上面的數據：

  • 工具效果取決於題型，不是越多越好。結構題（case-1/4/13）靜態索引贏 grep；字串題（case-5/6）grep 就滿分，加索引浪費。混用甚至有害（V4 餵太多上下文 → case-7 想錯）。
  • 選工具的決策要放程式層，不是 prompt。V3 賭模型自律失敗；V4 程式按題型派工具才把找檔分數穩定拉起來。
  • 少數寫法靜態無解，改 code 才是根本解。case-7、case-3 這類「答案不在靜態結構裡」的，所有工具所有模型都低分；改寫後（case-3 已實測）立刻見效。而且改 code 的好處對 grep、索引、所有模型通用，不會因為下個模型發布而過時。

一個反直覺的點

「名字是變數的動態寫法都該退役」是錯的。public_send("do_#{state}!")（case-2）、method_missing（case-12）名字都動態，卻都接近滿分——因為有固定前綴可搜、或範圍局部。真正的弱項不是「動不動態」，是「能不能被靜態追蹤」：字串建構名稱 + 無固定前綴 + 跨檔才能反推，三者同時成立才無解。

方法學限制

  • n=1：每格只跑一次，單次變異大。所有分數是方向訊號，不是定論；±0.5 的個案差異不應過度解讀。
  • 單一 judge：判讀題由同一個評分者判，避免標準漂移，但仍是主觀。
  • 3 模型、15 case：樣本小。結論（題型分工、case-7 全崩、改 code 見效）方向夠強，但模型內個案需更多 N。
  • 單一 codebase 家族：case 取自同一組生產 Rails 專案，pattern 分布未必代表所有 Ruby 專案。
